1. 서비스 소개
   개요: 이미지 첨부가 가능한 개인형 사진 일기장 서비스입니다.

타겟 사용자: 본인의 소중한 일상을 기록하고 소장하고 싶은 모든 사용자.

주요 기능:

콘텐츠 관리 (CRUD): 이미지 업로드와 함께 일기 작성, 수정, 삭제 기능 제공.

유효성 검증: 업로드 시 이미지 포맷(확장자, MIME 타입) 및 용량 제한을 통한 시스템 안정성 확보.

데이터 익스포트 (Lv3): 주문 시 해당 게시글의 메타데이터와 원본 이미지를 하나의 ZIP 파일로 통합 추출하여 인쇄 공정 등에 활용 가능한 구조 제공.

2. 실행 방법 (Docker)
   심사관의 환경에서 즉시 실행 가능하도록 Docker 환경 설정을 완료했습니다.

Bash
# 1. 환경 변수 준비
cp .env.example .env

# 2. 실행 (빌드 포함)
docker-compose up --build

# 3. 접속
# 브라우저에서 http://localhost:8080 접속 (포트 변경 필요 시 docker-compose.yml 수정 가능)
3. 완성한 레벨
   Lv1. 콘텐츠 관리 (완료): 게시글 기본 CRUD 구현.

Lv2. 사진 업로드 (완료): 로컬 파일 시스템 저장 및 도커 볼륨 매핑을 통한 데이터 영속성 확보.

Lv3. 주문 데이터 익스포트 (완료): 주문 건에 대한 JSON 메타데이터 및 이미지 통합 ZIP 압축 생성.

4. 기술 스택 및 아키텍처
   Framework: Spring Boot 3.x - 빠른 프로토타이핑과 높은 생산성, 풍부한 생태계를 활용하기 위해 선택했습니다.

Language: Java 17 - Spring Boot와의 호환성과 타입 안정성을 위해 선택했습니다.

Database: H2 Database (File Mode) - 별도의 DB 컨테이너 분리 없이도 가볍게 구동되며, 파일 모드를 통해 컨테이너 재시작 시에도 데이터가 유지되는 영속성을 구현했습니다.

View Engine: Mustache - 서버 사이드 렌더링을 위한 가볍고 직관적인 템플릿 엔진으로, 로직과 뷰를 깔끔하게 분리할 수 있어 선택했습니다.

5. AI 도구 사용 내역 (Q5 관련)
   본 프로젝트는 Gemini를 기술적 파트너로 적극 활용하여 개발 효율을 극대화했습니다.

설계 보조: 전체적인 서비스 아키텍처 설계 및 도커 인프라 구조 제안.

코드 최적화: 파일 입출력 로직에서 현대적인 java.nio.file.Path API 적용 및 예외 처리 가이드라인 확보.

문제 해결: 도커 볼륨 매핑 시 발생하는 경로 인식 문제(404 에러) 분석 및 H2 파일 모드 사용 시 데이터 중복 방지(MERGE INTO) 로직 도출.

6. 설계 의도 및 소회
   서비스 선택 이유: 스위트북의 인쇄 API 비즈니스를 가장 효과적으로 보여줄 수 있는 형태는 "사용자의 콘텐츠가 쌓이는 서비스"라고 판단했습니다. 사진 일기는 텍스트와 멀티미디어가 결합된 가장 기본적인 콘텐츠 데이터셋입니다.

사업성 판단: 최근 'Setlog'와 같이 개인의 일상을 Vlog 형태로 담는 SNS 트렌드에 주목했습니다. 단순히 휘발되는 기록이 아니라, 이를 잘 정제하여 인쇄물이나 디지털 자산으로 변환하는 모델은 충분한 사업성이 있다고 판단했습니다.

향후 과제: 개발 기간의 제약으로 구현하지 못한 '회원별 기록 관리(Multi-tenancy)' 기능과 '회원 정보 보안을 위한 비밀번호 Hash 도입' 등을 추후 고도화하고 싶습니다. 특히 보안 영역은 실제 상용 서비스로 나아가기 위한 필수 요소로 인식하고 있습니다.
